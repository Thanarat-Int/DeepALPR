# 📋 วิเคราะห์ Requirement & เตรียมประชุม — โปรเจกต์ Deep ALPR

> เอกสารนี้ทำขึ้นเพื่อเตรียมเข้าประชุมกับลูกค้า จากบรีฟที่ได้รับ (ข้อความ 4 บรรทัด)
> เป้าหมาย: เปลี่ยน "ข้อความสั้นๆ" ให้เป็น "requirement ที่ออกแบบและทำสัญญาได้จริง"

---

## ⚡ สรุปสำหรับคนรีบ (อ่านอันเดียวพอ)

1. บรีฟที่ได้มี **4 บรรทัด** — ยังขาดข้อมูลที่จำเป็นต่อการออกแบบ/เสนอราคา ~80%
2. **อย่า commit ราคาหรือยืนยันตัวเลข 95% ในที่ประชุม** ก่อนเคลียร์ scope ให้จบ
3. ตัวเลข "≤35 km/h" บอกใบ้ชัดว่านี่คืองาน **Access Control (ทางเข้า-ออก)** ไม่ใช่ traffic surveillance
4. เป้าหมายการประชุม = **ถามให้ครบ** ไม่ใช่เสนอให้ครบ — ใครถามเก่งกว่าคุมงานได้
5. คุณมี **PoC ที่รันได้จริงอยู่แล้ว** (โปรเจกต์นี้) — เอาไปเปิดโชว์เป็นตัวตั้งต้นการคุยได้เลย

---

## 1. บรีฟบอกอะไร และ "ไม่" บอกอะไร

| สิ่งที่บรีฟบอก | ช่องโหว่ที่ต้องถาม |
|----------------|---------------------|
| รองรับป้ายไทย "ทุกประเภท" | ทุกประเภทจริงไหม? ป้ายประมูล/กราฟิก, มอเตอร์ไซค์, ป้ายแดง, ทูต — อ่านยากมาก |
| Accuracy ≥ 95% | วัดยังไง? นับทั้งป้ายหรือรายตัวอักษร? เงื่อนไขแสง/มุม/สภาพป้ายแบบไหน? |
| จับรถ ≤ 35 km/h ได้ > 95% | แล้วรถเร็วกว่า 35 ล่ะ? วัดความเร็วด้วยอะไร? |
| มี API ส่ง ทะเบียน/เวลา/รูป | ส่งให้ระบบอะไร? ใครเป็นคนรับ? real-time หรือ batch? |

**สิ่งที่บรีฟไม่ได้บอกเลย (แต่จำเป็นต้องรู้):**
ใช้ที่ไหน · กี่กล้อง/กี่เลน · มีกล้องเดิมอยู่แล้วไหม · ควบคุมไม้กั้นด้วยไหม ·
มี dashboard/หน้าจอคนใช้ไหม · ใครดูแลระบบ · ใครซื้อ hardware · งบ · timeline ·
กลางคืน/ฝนต้องได้ไหม · ปริมาณรถต่อวัน

## 2. การตีความบรีฟ: นี่คืองาน "Access Control" ไม่ใช่ "Traffic Surveillance"

**หลักฐาน:** ข้อกำหนด "ความเร็วไม่เกิน 35 km/h" — นี่คือความเร็วที่รถวิ่งผ่าน
*ทางเข้า-ออกที่มีการควบคุม* ถ้าเป็นงานติดตามจราจรบนถนน/ทางหลวง จะไม่มีเพดาน 35

**Use case ที่เป็นไปได้สูง (ถามให้ชัดข้อแรกในที่ประชุม):**
- 🏘️ คอนโด / หมู่บ้านจัดสรร — แยกลูกบ้าน/ผู้มาเยือน, เปิดไม้กั้นอัตโนมัติ
- 🅿️ ลานจอดรถ / อาคารสำนักงาน — บันทึกเข้า-ออก, คิดค่าจอด
- 🏭 โรงงาน / นิคมอุตสาหกรรม — บันทึกรถเข้า-ออก, รักษาความปลอดภัย
- 🚧 ด่าน / จุดตรวจ — บันทึกทะเบียนรถผ่าน

> **คำถามแรกสุดของการประชุม:** "ระบบนี้จะติดตั้งที่ไหน และใช้ทำอะไร?"
> คำตอบข้อนี้เปลี่ยนทุกอย่าง — สถาปัตยกรรม, hardware, ราคา, ความเสี่ยง

## 3. ⭐ คำถามที่ต้องถามในที่ประชุม (เอาไปใช้ได้เลย)

### 3.1 Use case & สถานที่
- ติดตั้งที่ไหน? ใช้ทำอะไร? (เข้า-ออกหมู่บ้าน / ลานจอด / โรงงาน / อื่นๆ)
- กี่จุดติดตั้ง? กี่เลน/กี่กล้องต่อจุด? ทิศทางเดียวหรือเข้า-ออก?
- ปริมาณรถต่อวันประมาณเท่าไหร่?
- กลางวันอย่างเดียว หรือ 24 ชม.? กลางแจ้งหรือในร่ม?

### 3.2 ขอบเขตงาน (สำคัญที่สุดต่อราคา)
- ลูกค้าอยากได้ **เฉพาะซอฟต์แวร์** หรือ **ซอฟต์แวร์ + hardware + ติดตั้ง (turnkey)?**
- มีกล้อง/คอมพิวเตอร์/ไม้กั้น อยู่แล้วหรือต้องจัดหาใหม่?
- ใครรับผิดชอบ: เดินสาย, ไฟ, งานโยธา, internet หน้างาน?

### 3.3 กล้อง & Hardware
- มีกล้องอยู่แล้วไหม? ยี่ห้อ/รุ่นอะไร? (กล้องทั่วไป ≠ กล้อง LPR)
- ถ้าจะใช้กล้องเดิม — ขอภาพ/วิดีโอตัวอย่างจากกล้องจริงมาทดสอบ
- ระยะ/มุม/ความสูงที่ติดกล้องได้ — มีข้อจำกัดไหม?
- มีไฟส่องสว่าง/IR สำหรับกลางคืนไหม?

### 3.4 การทำงาน (Workflow)
- ระบบแค่ "อ่านแล้วบันทึก" หรือต้อง **ควบคุมไม้กั้น/ประตู** ด้วย?
- ต้องเทียบกับ **ฐานข้อมูลรถที่ลงทะเบียน** ไหม? (whitelist ลูกบ้าน / blacklist)
- ถ้าอ่านไม่ออก หรือเป็นรถแปลกหน้า — ให้ทำอะไร? (แจ้ง รปภ.? เปิดมือ?)
- ต้องมีระบบลงทะเบียนรถ/ผู้มาเยือนไหม?

### 3.5 Interface / Dashboard ← (สิ่งที่คุณยังไม่แน่ใจ)
- ต้องมี **หน้าจอให้เจ้าหน้าที่/รปภ.** ดูแบบ real-time ไหม?
- ต้องมี **หน้าผู้ดูแลระบบ** (จัดการรถที่ลงทะเบียน, ออกรายงาน) ไหม?
- ต้องการ **รายงาน** อะไรบ้าง? ส่งให้ใคร? ความถี่?
- ดูผ่านเว็บ / แอป / จอเฉพาะ? กี่คนใช้พร้อมกัน?

### 3.6 API & Integration
- API นี้จะ "ส่งเข้า" ระบบอะไรของลูกค้า? (ระบบนิติบุคคล / parking / ERP / security)
- ลูกค้ามีเอกสาร spec ของระบบปลายทางไหม?
- ต้องการแบบ push (ยิงทันที) หรือ pull (ให้เขาดึง)?
- ส่งรูปแบบไหน — URL รูป หรือไฟล์ base64?

### 3.7 Accuracy & การตรวจรับงาน (UAT)
- "95%" นับยังไง — ทั้งป้ายถูกหมด หรือนับรายตัวอักษร?
- วัดจากกลุ่มตัวอย่างกี่คัน? เก็บช่วงเวลาไหน?
- เงื่อนไขที่ใช้วัด: ป้ายสะอาด/ไม่บัง, รถอยู่ในโซน, ความเร็ว ≤35, แสงปกติ — ตกลงกันให้ชัด
- ป้ายสกปรก/งอ/บัง/ป้ายกราฟิก — รวมในการวัดไหม หรือยกเว้น?
- ใครเป็นคนตัดสินผ่าน/ไม่ผ่าน?

### 3.8 Data & กฎหมาย
- ข้อมูลเก็บที่ไหน — เครื่องหน้างาน (on-prem) หรือ cloud?
- เก็บนานเท่าไหร่? ใครเข้าถึงได้?
- ลูกค้ารู้เรื่อง **PDPA** ไหม? (ป้ายทะเบียน + รูปรถ = ข้อมูลส่วนบุคคล มีภาระตามกฎหมาย)

### 3.9 Operations & การดูแลหลังส่งมอบ
- ใครดูแลระบบหลังติดตั้ง? มีสัญญา support/MA ไหม?
- ถ้ากล้องเสีย/เน็ตหลุด/ไฟดับ — ใครรับผิดชอบ?
- ต้องมีการ train ผู้ใช้งานไหม?

### 3.10 Commercial
- งบประมาณคร่าวๆ? — ถามตรงๆ ได้ จะได้เสนอให้ตรงระดับ
- Timeline ที่ต้องการ? มี deadline อะไรผูกไหม?
- รูปแบบจ่าย: เหมาจ่ายครั้งเดียว / รายเดือน / มี MA แยก?

## 4. ระบบ ALPR ระดับ Production ทำงานยังไง (ครบวงจร)

### 4.1 ชั้น Hardware หน้างาน
```
   รถเข้า
     │
 [Trigger] ──► loop ฝังพื้น / radar / หรือใช้ video วิเคราะห์เอง
     │
  [กล้อง LPR] ──► ต้อง shutter เร็วพอหยุดภาพรถที่ 35 km/h (~1/1000s)
     │            + WDR (สู้แสงย้อน) + IR สำหรับกลางคืน
     │            *** กล้อง CCTV ทั่วไปมักไม่พอ — ต้องกล้องสำหรับ LPR ***
     │
 [ไฟ IR/LED] ──► กลางคืน shutter เร็ว = ภาพมืด ต้องมีไฟช่วย
     │
 [Edge box] ──► mini PC / NVIDIA Jetson ที่หน้างาน ประมวลผล real-time
     │
 [ไม้กั้น] ──► ถ้าต้องคุมประตู: เชื่อมผ่าน relay / dry contact
     │
 [UPS + Network] ──► กันไฟดับ + เชื่อมกลับ server
```

> **จุดที่คนมองข้ามบ่อย:** ตำแหน่ง/มุม/ระยะของกล้อง สำคัญกว่าตัว AI
> ถ้ากล้องติดมุมไม่ดี ป้ายเอียง/เล็ก/ย้อนแสง — AI เก่งแค่ไหนก็ช่วยไม่ได้
> และที่ 35 km/h รถวิ่ง ~10 เมตร/วินาที — ถ้า shutter ไม่เร็วพอ ภาพเบลอ อ่านไม่ออก

### 4.2 ชั้น Software (pipeline)
```
เฟรมวิดีโอ → ตรวจจับป้าย → ติดตาม + วัดความเร็ว → เกตความเร็ว ≤35
   → OCR อ่านทะเบียน → โหวตหลายเฟรม → ได้ทะเบียนที่มั่นใจ
   → เทียบฐานข้อมูล (ลูกบ้าน/blacklist) → ตัดสินใจ (เปิดไม้กั้น/แจ้งเตือน)
   → บันทึก + ส่ง API/webhook
```
*(นี่คือสิ่งที่ PoC ของเราทำได้แล้ว ยกเว้นส่วน "เทียบฐานข้อมูล" และ "คุมไม้กั้น")*

### 4.3 ชั้น Data
- **ฐานข้อมูลรถที่ลงทะเบียน** — ลูกบ้าน/พนักงาน/สมาชิก (whitelist), บัญชีดำ (blacklist)
- **Log เหตุการณ์** — ทุกคันที่ผ่าน: ทะเบียน, เวลา, รูป, เข้า/ออก, ผลการตัดสิน
- **Sync** — เครื่องหน้างานเก็บ local + ส่งขึ้นส่วนกลาง

### 4.4 ตัวอย่าง Workflow จริง (กรณีหมู่บ้าน)
1. รถเข้าใกล้ → trigger จับได้
2. กล้องถ่าย → ALPR อ่านทะเบียน
3. ระบบเทียบฐานข้อมูล:
   - **ลูกบ้าน** → เปิดไม้กั้นอัตโนมัติ + บันทึก
   - **ผู้มาเยือนลงทะเบียนล่วงหน้า** → เปิดอัตโนมัติ + บันทึก
   - **ไม่รู้จัก** → เด้งแจ้งบนจอ รปภ. → รปภ. ตรวจสอบ → เปิดมือ
4. ทุกเหตุการณ์ → log + ยิง event ผ่าน API ไประบบนิติบุคคล
5. ออกรายงานประจำวัน/เดือน

## 5. Interface / Dashboard — ต้องมีไหม? มีอะไรบ้าง?

**ตอบตรงๆ: งาน access control ระดับ production เกือบทั้งหมด "ต้องมี"** — API อย่างเดียวไม่พอ
เพราะมีคนต้องใช้งาน (รปภ., ผู้ดูแล) สิ่งที่ production มักต้องมี:

| ส่วน | ใครใช้ | ทำอะไร |
|------|--------|--------|
| **Operator Console** | รปภ./เจ้าหน้าที่ | ดู real-time, เห็นรถเข้า, กดเปิดไม้กั้นเอง, ค้นหา |
| **Admin Panel** | ผู้ดูแลระบบ | จัดการรถที่ลงทะเบียน, whitelist/blacklist, ผู้ใช้, ตั้งค่า |
| **Reports** | ผู้บริหาร/นิติ | รายงานรถเข้า-ออก, สถิติ, เคสผิดปกติ |
| **REST API** | ระบบอื่นของลูกค้า | ดึง/รับข้อมูลไปใช้ต่อ |
| (ทางเลือก) Mobile | ลูกบ้าน/ผู้บริหาร | ลงทะเบียนรถผู้มาเยือน, ดูแจ้งเตือน |

**ระดับ scope (เอาไว้เสนอลูกค้าเป็นทางเลือก):**
- **ขั้นต่ำ:** API + หน้า dashboard ดูผลอย่างเดียว
- **มาตรฐาน:** + Operator console + Admin panel + รายงานพื้นฐาน
- **เต็มรูปแบบ:** + คุมไม้กั้น + ระบบลงทะเบียนผู้มาเยือน + mobile + integration

> 💡 PoC ที่เราสร้างไว้มี dashboard ดูผล + REST API + webhook แล้ว = "ขั้นต่ำ" พร้อมโชว์

## 6. ⚠️ กับดักของ "Accuracy 95%" — อ่านก่อนเซ็นสัญญา

ตัวเลข 95% ในสัญญา = **ตัวเลขที่ผูกพันทางกฎหมาย** ถ้าไม่ระบุเงื่อนไข ลูกค้าปฏิเสธรับงานได้
แม้ระบบจะดี เพียงเพราะมันพลาดป้ายสกปรก/ตอนกลางคืน/มุมแปลกๆ

**ต้องเขียนในสัญญาให้ชัด — "95% ภายใต้เงื่อนไข":**
- วัดแบบไหน (ทั้งป้ายถูกหมด แนะนำให้นิยามให้ชัด)
- วัดจากรถกี่คัน เก็บช่วงไหน ใครเก็บ
- **Operational Design Domain (ODD)** — ขอบเขตที่การันตี:
  - ป้ายมาตรฐาน สะอาด ไม่ชำรุด ไม่ถูกบัง
  - รถอยู่ในโซนจับภาพที่กำหนด + ความเร็ว ≤ 35 km/h
  - สภาพแสงตามที่ตกลง (ถ้าต้องได้กลางคืน ต้องมีไฟ IR)
- สิ่งที่ **อยู่นอกการการันตี** — ป้ายกราฟิก/ประมูล, ป้ายงอ/เลือน, ป้ายต่างชาติ, รถซ้อนคัน

> ถ้าลูกค้ายืนยัน "ทุกประเภท ทุกสภาพ 95%" แบบไม่มีเงื่อนไข — นั่นคือ **ธงแดง**
> ไม่มีระบบ ALPR ไหนในโลกการันตีแบบนั้นได้

## 7. สร้างเอง vs ใช้ Engine สำเร็จรูป

| ทาง | เหมาะเมื่อ |
|-----|-----------|
| **ใช้ ALPR engine เชิงพาณิชย์** (เช่น Plate Recognizer) | อยากขึ้นงานเร็ว, ถึง 95% ชัวร์, ความเสี่ยงต่ำ — **แนะนำสำหรับงานส่งลูกค้า** |
| **กล้อง ANPR สำเร็จรูป** | ลูกค้ายอมซื้อกล้องเฉพาะ, ลดงานเขียน CV |
| **เทรนโมเดลเอง** (แบบ PoC นี้) | ต้องเป็นเจ้าของ IP, ปริมาณเยอะจนค่า license แพง, มี data |

**คำแนะนำ:** value ที่คุณขายได้จริงคือ **การ integrate + workflow + dashboard + การติดตั้ง**
ไม่ใช่ตัวโมเดล OCR — เอาเวลาไปลงกับส่วนที่ลูกค้าเห็นค่า

## 8. ข้อเสนอ: ทำเป็นเฟส (เอาไปเสนอในที่ประชุม)

| เฟส | งาน | ทำไม |
|-----|-----|------|

| **1. PoC** | พิสูจน์ความแม่นบนวิดีโอจริงของลูกค้า 1 จุด | ลดความเสี่ยงทั้งสองฝ่ายก่อน commit ใหญ่ |
| **2. Pilot** | ติดตั้งจริง 1 เลน ใช้งานจริง | ตรวจรับ 95% บนของจริง |
| **3. Rollout** | ขยายทุกจุด | — |
| **4. MA / Support** | ดูแลรายปี | รายได้ต่อเนื่อง |

## 9. 🚩 ธงแดง / ความเสี่ยงที่ต้องระวัง

- "ทุกประเภท + 95%" แบบไม่มีเงื่อนไข → ปฏิเสธ หรือขอใส่ ODD
- ลูกค้าจะใช้ **กล้อง CCTV เดิม** ที่ไม่เหมาะกับ LPR → ความแม่นจะตก ต้องเตือนแต่แรก
- ขอบเขต hardware ไม่ชัด → ใครซื้อกล้อง/เดินไฟ/ไม้กั้น = เงินก้อนใหญ่
- Dashboard scope บานปลาย → ตกลง scope หน้าจอให้ชัดเป็นลายลักษณ์
- กลางคืน/ฝน → ถ้าลูกค้าต้องการ ต้องมี IR + ยอมรับว่าแม่นน้อยลง
- PDPA → ถ้าลูกค้าไม่คิดเรื่องนี้ ความรับผิดอาจตกมาที่คุณ
- ระบบปลายทางของ API ยังไม่นิ่ง → integration จะวุ่น ขอ spec ให้ได้

## ภาคผนวก: PoC ที่มีอยู่ใช้พูดอะไรได้บ้าง

โปรเจกต์นี้ (PoC) **พิสูจน์แล้วว่า pipeline ทำงานได้:** อ่านป้ายไทย, เกตความเร็ว,
REST API + webhook, dashboard, OCR แม่น 99.6% บนข้อมูลสังเคราะห์

**พูดกับลูกค้าได้ว่า:** "นี่คือ proof-of-concept ที่ผมทำไว้แล้ว แสดงว่าแนวทางเป็นไปได้
ขั้นต่อไปคือเก็บข้อมูลจากหน้างานจริงเพื่อปรับให้ตรงสภาพแวดล้อมของคุณ"

**อย่าพูดเกินจริงว่า:** ระบบพร้อมใช้ 100% แล้ว — เพราะ data เป็นของสังเคราะห์
ยังต้องเทรน/วัดผลใหม่ด้วยภาพจริง
