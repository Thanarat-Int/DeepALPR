เช็คลิสต์ก่อน Go-Live 10 ข้อ
============================

[ ] 1. เก็บข้อมูลจริงและเทรนโมเดลใหม่
       เป้าหมาย: ความแม่นยำ OCR ถึง 95% บนป้ายจริงจากกล้องที่ใช้งานจริง
       งานที่ต้องทำ:
         ก. ติดตั้งกล้องทดสอบที่ไซต์
         ข. เก็บภาพต่อเนื่อง 2 สัปดาห์ ครอบคลุมเช้า บ่าย กลางคืน และฝน
         ค. ตัดภาพป้ายทะเบียนออกมา 5,000 ถึง 20,000 ภาพ
         ง. ใส่ ground truth ตัวอักษร (ทำเองหรือจ้างที่ราคา 3 ถึง 7 บาทต่อภาพ)
         จ. Fine-tune CRNN จาก checkpoint เดิมที่เทรนกับ synthetic
         ฉ. ทดสอบกับชุดข้อมูลจริงที่กันไว้ แล้วทำซ้ำจนถึง 95%
       เวลาที่ใช้       : 3 ถึง 4 สัปดาห์
       ขั้นจำเป็น       : ใช่ ขาดไม่ได้ก่อน go-live


[ ] 2. ติดตั้งฮาร์ดแวร์ที่ไซต์งาน
       อุปกรณ์ที่ต้องมีต่อหนึ่งช่องทาง:
         ก. กล้อง IP ความละเอียดอย่างน้อย 2 ล้านพิกเซล มีโหมดกลางคืน
            และไฟ IR เลนส์ 6 ถึง 12 มม.
         ข. กล่องคอมพิวเตอร์ edge: Jetson Orin Nano หรือ Intel NUC
            ที่มี GPU หรือ TPU
         ค. กล่องควบคุมไม้กั้นที่รองรับ Wiegand, RS485 หรือ MQTT
         ง. สาย LAN กิกะบิต ไม่ใช้ Wi-Fi กับกล้อง
         จ. UPS สำรองไฟอย่างน้อย 30 นาที
         ฉ. SSD 256 GB ขึ้นไป สำหรับเก็บข้อมูลที่ไซต์ + NAS ส่วนกลางเก็บถาวร
       เวลาที่ใช้       : 1 สัปดาห์ต่อช่องทาง
       ขั้นจำเป็น       : ใช่ ขาดไม่ได้


[ ] 3. เปลี่ยนฐานข้อมูลเป็นระดับ production
       ย้ายจาก SQLite (POC) ไปเป็น PostgreSQL หรือ MySQL
       งานที่ต้องทำ:
         ก. จัดเตรียมเซิร์ฟเวอร์ฐานข้อมูล (managed service หรือ self-host)
         ข. ปรับ SQL syntax และ connection pooling ในโค้ด
         ค. ตั้ง backup รายคืน เก็บออกนอกไซต์ และทดสอบ restore
         ง. เพิ่ม read replica ถ้ามีหลายช่องทาง
       เวลาที่ใช้       : 3 ถึง 5 วัน
       ขั้นจำเป็น       : ควรทำ ไม่ blocker ถ้ามีช่องทางเดียว


[ ] 4. เพิ่มความปลอดภัยระดับ production
       งานที่ต้องทำ:
         ก. ติดตั้ง HTTPS ผ่าน Nginx หรือ Caddy ใช้ Let's Encrypt
         ข. เก็บ session token ใน Redis แทน in-memory หรือใช้ JWT secret
         ค. ตั้ง rate limit ต่อ IP และต่อ API key
         ง. หมุน API key และลบบัญชี demo (admin/admin123, operator/operator123)
         จ. ตั้ง firewall ให้เปิดเฉพาะ HTTPS ออกสาธารณะ
       เวลาที่ใช้       : 3 วัน
       ขั้นจำเป็น       : ใช่ ขาดไม่ได้


[ ] 5. ต่อระบบกับไม้กั้นจริง
       ปัจจุบัน /api/gate/open แค่ log การเปิด (จำลอง)
       งานที่ต้องทำ:
         ก. เลือกโปรโตคอลที่จะใช้ (Wiegand, RS485, MQTT หรือ HTTP webhook)
         ข. เขียน driver ในฝั่ง backend และต่อเข้ากับบอร์ดรีเลย์
         ค. ใส่ fail-safe ถ้าฮาร์ดแวร์ตอบไม่ได้ ต้องแจ้งเตือน operator
            ไม่ใช่เงียบหายไป
         ง. ทดสอบ end-to-end กับไม้กั้นจริง
       เวลาที่ใช้       : 4 ถึง 7 วัน
       ขั้นจำเป็น       : ใช่ ขาดไม่ได้


[ ] 6. Monitoring, Logging และ Backup
       งานที่ต้องทำ:
         ก. ตั้ง Prometheus + Grafana แสดง API latency, error rate, FPS ของ pipeline
         ข. รวม log ที่เดียว (Loki, ELK หรือ CloudWatch)
         ค. ตั้ง alert ผ่าน PagerDuty, LINE Notify หรืออีเมล ในกรณี:
            กล้องดับ, OCR confidence ตก, ไม้กั้นเสีย, DB หลุด
         ง. ทดสอบ restore backup ของฐานข้อมูลและคลังภาพให้ใช้ได้จริง
       เวลาที่ใช้       : 4 ถึง 6 วัน
       ขั้นจำเป็น       : ควรทำ


[ ] 7. PDPA และข้อกฎหมาย
       ทะเบียนรถ ข้อมูลเจ้าของ และภาพถ่ายนับเป็นข้อมูลส่วนบุคคลตาม PDPA
       งานที่ต้องทำ:
         ก. ติดป้ายแจ้ง privacy notice ทุกทางเข้าที่มีกล้อง
         ข. มีเอกสารยินยอม (consent) สำหรับทุกคันที่ลงทะเบียนใน whitelist
         ค. มีนโยบาย retention เขียนเป็นเอกสาร (ระบบเก็บ 90 วันอยู่แล้ว)
         ง. แต่งตั้ง DPO (Data Protection Officer) ถ้าไซต์ใหญ่พอ
         จ. ตรวจสอบกฎหมาย CCTV ของท้องที่
         ฉ. เช็ค license ของ Ultralytics YOLOv8: เป็น AGPL-3.0
            ถ้าใช้เชิงพาณิชย์ต้องเปิด source หรือซื้อ commercial license
            จาก Ultralytics
       เวลาที่ใช้       : 1 ถึง 2 สัปดาห์ (รอ legal review)
       ขั้นจำเป็น       : ใช่ ขาดไม่ได้


[ ] 8. SLA และเอกสารปฏิบัติงาน
       ตกลงกับลูกค้าและเขียนเป็นเอกสาร:
         ก. เป้า uptime (แนะนำ 99.5%)
         ข. ความแม่นยำ OCR ขั้นต่ำ (จาก brief: 95%)
         ค. Latency end-to-end ตั้งแต่ detect ถึง gate open (เป้าน้อยกว่า 500 ms)
         ง. เวลา response เมื่อเกิด incident และ escalation path
         จ. หน้าต่างเวลาที่ทำ maintenance ได้
         ฉ. Runbook สำหรับเหตุที่เจอบ่อย: กล้องดับ, OCR drift, ไม้กั้นค้าง,
            DB locked, ดิสก์เต็ม
       เวลาที่ใช้       : 3 ถึง 5 วัน
       ขั้นจำเป็น       : ควรทำ


[ ] 9. Pilot run แบบ shadow mode
       รัน 4 สัปดาห์ก่อนเปิดสวิตช์จริง
       งานที่ต้องทำ:
         ก. ระบบตัดสินใจได้แต่ "ไม่" สั่งไม้กั้นจริง
         ข. log ทุก decision และให้ operator เทียบกับของจริงทุกวัน
         ค. วัด real-world accuracy, false-grant rate, false-deny rate
         ง. ปรับ threshold เช่น confidence cutoff, ความเข้มของการ match ป้าย
         จ. รับเซ็นรับรองจากลูกค้าก่อน go-live
       เวลาที่ใช้       : 4 สัปดาห์ตามปฏิทิน ใช้เวลา ops วันละประมาณ 2 ชั่วโมง
       ขั้นจำเป็น       : ควรทำ


[ ] 10. Go-Live และ support ที่ไซต์
       งานที่ต้องทำ:
         ก. แผน cutover พร้อมขั้นตอน rollback ชัดเจน
         ข. มีวิศวกรประจำที่ไซต์ 2 สัปดาห์แรก
         ค. ประชุมรีวิวรายวันสัปดาห์แรก
         ง. ประชุมรีวิวรายสัปดาห์ตลอดเดือนแรก
         จ. นัด retrain โมเดลครั้งแรกที่เดือน 3
         ฉ. นัดเช็ดเลนส์กล้องและปรับมุมทุกเดือน
       เวลาที่ใช้       : 2 สัปดาห์เข้ม + ต่อเนื่อง
       ขั้นจำเป็น       : ตัว go-live เอง
